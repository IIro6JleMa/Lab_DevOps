# Лабораторный стенд CI/CD и DevSecOps на локальной инфраструктуре

## Описание

В данном репозитории представлен локальный учебный стенд, демонстрирующий базовый процесс **CI/CD** и элементы **DevSecOps** с использованием только локально разворачиваемых инструментов.

В рамках стенда:

- поднимается локальный **GitLab**;
- создаётся локальный Kubernetes-кластер на базе **k3d/k3s**;
- используется локальный **Docker Registry**;
- разворачивается тестовое микросервисное приложение;
- настраивается **GitLab CI/CD pipeline**;
- выполняется автоматическая сборка, сканирование, публикация образов и деплой в Kubernetes.

Стенд предназначен для учебных целей и может использоваться как шаблон для лабораторной работы по дисциплинам, связанным с DevOps, CI/CD, контейнеризацией и основами DevSecOps.

---

## Цель лабораторной работы

Развернуть полностью локальный стенд CI/CD и DevSecOps, включающий:

- локальный GitLab;
- локальный Kubernetes-кластер;
- локальный registry контейнерных образов;
- GitLab Runner;
- тестовое микросервисное приложение;
- pipeline со стадиями сборки, сканирования, публикации и деплоя.

Результатом выполнения работы должен быть работающий стенд, в котором после `git push` в ветку `main` приложение автоматически собирается, проходит базовую проверку, публикуется в локальный registry и разворачивается в Kubernetes.

---

## Архитектура стенда

В состав стенда входят следующие компоненты.

### 1. GitLab CE

Используется как локальный Git-сервер, система хранения репозиториев и CI/CD платформа.

### 2. Local Docker Registry

Используется для хранения контейнерных образов, которые собираются в pipeline.

### 3. k3d / k3s

Используется для развёртывания локального Kubernetes-кластера.

### 4. GitLab Runner

Исполняет pipeline и выполняет:

- сборку контейнеров;
- сканирование проекта;
- публикацию образов;
- деплой в Kubernetes.

### 5. Тестовое приложение

Состоит из двух сервисов:

- `api` — backend на FastAPI с endpoint `/health`;
- `frontend` — frontend на Nginx, который отдаёт простой текстовый ответ и проксирует `/api/` на backend.

## Минимальные требования к машине

Рекомендуемые параметры:

- CPU: от 4 ядер
- RAM: от 8 ГБ
- Диск: от 15–20 ГБ свободного места

Если GitLab работает медленно, желательно использовать 12–16 ГБ оперативной памяти.

## Сетевые параметры и важное замечание

В текущих конфигурациях используется IP-адрес `192.168.237.135`.

Это IP-адрес машины, на которой запускаются:

- GitLab
- Docker Registry
- k3d / k3s
- GitLab Runner

Перед выполнением лабораторной необходимо определить IP своей машины и при необходимости заменить `192.168.237.135` на свой адрес во всех конфигурационных файлах.

Узнать IP можно командами `hostname -I` или `ip a`.

Для удобства далее в инструкции используется переменная `HOST_IP=192.168.237.135`.

Если у вас другой адрес, подставьте свой.

## Шаг 1. Поднять локальный GitLab

Для запуска GitLab используется файл `platform/docker-compose.yml`.

Нужно перейти в каталог `platform` и выполнить команду `docker compose up -d`.

После запуска контейнера необходимо проверить, что GitLab действительно стартовал. Для этого можно использовать команды `docker ps` и `docker logs -f gitlab`.

Первый запуск GitLab может занимать от 10 до 20 минут.

После запуска GitLab будет доступен по адресу `http://<HOST_IP>:8929`.

Например: `http://192.168.237.135:8929`.

---

## Шаг 2. Получить пароль администратора GitLab

После первого запуска GitLab нужно получить начальный пароль пользователя `root`.

Для этого выполняется команда `docker exec -it gitlab grep 'Password:' /etc/gitlab/initial_root_password`.

Из вывода нужно скопировать пароль и войти под пользователем:

- логин: `root`
- пароль: из файла `initial_root_password`

После входа рекомендуется сразу изменить пароль на удобный.

---

## Шаг 3. Создать проект в GitLab

После входа в GitLab необходимо:

- создать новый проект;
- задать ему имя, например `test-project`;
- выбрать создание пустого проекта;
- сохранить URL проекта для дальнейшей отправки исходного кода.

Если работа ведётся локально, можно использовать либо HTTP URL, либо SSH URL. Для учебного стенда проще использовать HTTP, если не настроены SSH-ключи.

---

## Шаг 4. Поднять локальный Docker Registry

Для pipeline нужен локальный registry, в который будут публиковаться Docker-образы.

Пример запуска registry:

`docker run -d --restart=always -p 5001:5000 --name registry registry:2`

Проверить работу registry можно командой:

`curl http://<HOST_IP>:5001/v2/_catalog`

Если registry поднят, он должен вернуть пустой список репозиториев или список уже загруженных образов.

---

## Шаг 5. Подготовить файл registries.yaml

Для того чтобы k3d/k3s мог работать с локальным insecure registry, используется файл `registries.yaml`.

В нём описывается зеркало и настройки TLS для локального registry.

В данном проекте используется registry по адресу `<HOST_IP>:5001`.

Важно: если IP-адрес вашей машины отличается, его нужно заменить и в этом файле.

---

## Шаг 6. Создать Kubernetes-кластер через k3d

После запуска registry необходимо создать локальный кластер k3d с подключением к registry.

Пример логики действий:

- создать кластер с именем, например, `devops-lab`;
- подключить registry;
- передать `registries.yaml`;
- при необходимости пробросить 80 порт для ingress-контроллера.

Если используется Traefik по умолчанию, важно, чтобы ingress был доступен с хостовой машины.

После создания кластера необходимо проверить его доступность командой `kubectl get nodes`.

Если узлы отображаются, кластер готов к работе.

---

## Шаг 7. Проверить ingress-контроллер

Так как в проекте используется объект `Ingress` с `ingressClassName: traefik`, необходимо убедиться, что в кластере действительно работает Traefik.

Проверить это можно командой `kubectl get pods -A`.

Нужно убедиться, что поды Traefik находятся в состоянии `Running`.

---

## Шаг 8. Подготовить kubeconfig для GitLab CI/CD

Для того чтобы pipeline мог выполнять деплой в кластер, необходимо передать kubeconfig в GitLab CI/CD variables.

Нужно получить kubeconfig локального кластера и закодировать его в base64.

Логика следующая:

- получить kubeconfig через `k3d kubeconfig get <cluster-name>`;
- сохранить его во временный файл;
- закодировать содержимое в base64 без переносов строк;
- добавить полученное значение в переменные GitLab.

Важно понимать, что в текущем pipeline сервер Kubernetes дополнительно исправляется через `sed`, так как в kubeconfig может оказаться адрес, недоступный из контейнера runner'а.

---

## Шаг 9. Создать пользователя или токен для работы с GitLab

Для выполнения pipeline runner должен иметь доступ к проекту.

Обычно для этого достаточно:

- зарегистрировать runner в конкретный проект;
- использовать registration token проекта;
- при необходимости создать отдельного пользователя для учебного стенда.

Если используется push по HTTP, также может потребоваться personal access token или логин/пароль пользователя GitLab.

---

## Шаг 10. Поднять и зарегистрировать GitLab Runner

GitLab Runner можно поднять либо как контейнер Docker, либо установить как отдельный сервис.

Для учебного стенда проще использовать Docker-вариант.

Runner должен быть зарегистрирован в GitLab-проекте.

При регистрации runner'а важно выбрать executor, подходящий для сборки Docker-образов. В простом варианте обычно используют:

- Docker executor;
- privileged mode, если требуется Docker-in-Docker;
- либо shell executor, если runner установлен прямо на хосте и использует локальный Docker.

После регистрации нужно убедиться, что runner появился в интерфейсе GitLab и активен.

---

## Шаг 11. Подготовить CI/CD переменные в GitLab

В проекте необходимо создать переменную `KUBE_CONFIG`.

Она должна содержать base64-представление kubeconfig.

Переменная добавляется в GitLab через раздел:

`Settings -> CI/CD -> Variables`

Рекомендуется включить:
- `Protect variable` только если pipeline запускается в защищённых ветках;
- `Mask variable`, если формат значения это позволяет.

Если в дальнейшем потребуется авторизация в registry, можно также добавить дополнительные переменные, например для логина и пароля registry.

---

## Шаг 12. Разместить исходный код проекта в GitLab

После создания проекта и подготовки runner'а необходимо загрузить в GitLab исходный код из каталога `test-project`.

Общий порядок действий:

- перейти в каталог `test-project`;
- инициализировать git-репозиторий, если это ещё не сделано;
- добавить remote на GitLab-проект;
- выполнить commit;
- отправить код в ветку `main`.

После отправки кода в репозиторий GitLab автоматически запустит pipeline.

## Шаг 13. Что делает pipeline

Файл `.gitlab-ci.yml` описывает четыре стадии:

- `build`
- `scan`
- `push`
- `deploy`

### Стадия build

На стадии `build` собираются два контейнерных образа:

- backend-образ `hello-api`
- frontend-образ `hello-frontend`

Теги формируются на основе короткого SHA коммита.

### Стадия scan

На стадии `scan` используется Trivy.

Выполняются базовые проверки:

- `trivy fs` — сканирование файловой системы проекта;
- `trivy config` — проверка конфигураций.

Это не полноценный enterprise DevSecOps-контур, но для лабораторного стенда демонстрирует практику статической проверки инфраструктурных и проектных файлов.

### Стадия push

На стадии `push` образы пересобираются и публикуются в локальный registry.

Registry используется по адресу `<HOST_IP>:5001`.

### Стадия deploy

На стадии `deploy`:

- из переменной `KUBE_CONFIG` создаётся kubeconfig;
- адрес API-сервера Kubernetes подменяется на внешний IP хоста;
- создаётся namespace `devops-lab`;
- в Kubernetes применяются манифесты backend и frontend;
- создаётся Ingress;
- выполняется ожидание готовности deployment'ов.

Деплой настроен только для ветки `main`.

## Шаг 14. Kubernetes-манифесты

В проекте используются три основных файла.

### `k8s/api.yaml`

Описывает:

- Deployment для backend;
- Service для backend;
- readinessProbe на `/health`;
- livenessProbe на `/health`.

### `k8s/frontend.yaml`

Описывает:

- Deployment для frontend;
- Service типа `ClusterIP`.

### `k8s/ingress.yaml`

Описывает Ingress с хостом `hello.local`.

Это означает, что для доступа к приложению необходимо, чтобы имя `hello.local` резолвилось в IP-адрес вашей машины.

---

## Шаг 15. Настроить доступ по доменному имени hello.local

Так как Ingress использует `host: hello.local`, на локальной машине нужно прописать соответствие имени `hello.local` IP-адресу хоста.

На Linux это делается через файл `/etc/hosts`.

Нужно добавить строку вида:

`192.168.237.135 hello.local`

Если у вас другой IP, подставьте его.

После этого приложение станет доступно по адресу `http://hello.local`.

Если проброс порта 80 не выполнялся, может понадобиться использовать другой порт ingress-контроллера.

## Шаг 16. Проверка результата

После успешного pipeline необходимо проверить:

- что в GitLab pipeline завершился успешно;
- что в registry появились образы;
- что в Kubernetes создался namespace `devops-lab`;
- что deployment'ы `api` и `frontend` находятся в состоянии `Available`;
- что сервисы созданы;
- что Ingress создан;
- что приложение открывается по адресу `http://hello.local`.

Дополнительно можно проверить backend endpoint `/health` через frontend-прокси или напрямую изнутри кластера.

Для проверки состояния кластера и приложения удобно использовать следующие команды:

- `kubectl get nodes`
- `kubectl get ns`
- `kubectl get pods -n devops-lab`

## Возможные проблемы и способы их устранения

### 1. GitLab долго запускается

Это нормально для первого старта. Нужно подождать 10–20 минут и проверить логи контейнера.

### 2. Runner не выполняет Docker build

Возможные причины:

- runner зарегистрирован без нужного executor;
- runner не имеет доступа к Docker daemon;
- не включён privileged mode;
- не настроен Docker-in-Docker или shell executor.

### 3. Pipeline не может подключиться к Kubernetes

Возможные причины:

- неверный `KUBE_CONFIG`;
- kubeconfig был закодирован некорректно;
- в kubeconfig указан недоступный адрес API-сервера;
- runner не может достучаться до `<HOST_IP>`;
- сертификат Kubernetes не совпадает с адресом сервера.

В данном проекте часть этой проблемы обходится подменой адреса сервера и включением `insecure-skip-tls-verify`.

### 4. Kubernetes не может скачать образы из registry

Возможные причины:

- registry не доступен по сети;
- в `registries.yaml` указан неверный IP;
- кластер создан без подключения registry;
- образы не были успешно отправлены в registry.

### 5. Ingress не открывает приложение

Возможные причины:

- не прописан `hello.local` в `/etc/hosts`;
- не работает Traefik;
- не проброшен порт ingress наружу;
- frontend service недоступен;
- pods не вышли в `Running`.

### 6. Pipeline запускается, но deploy не происходит

Проверьте, что ветка называется именно `main`, так как deploy ограничен через `only: main`.

## Итоговый сценарий выполнения лабораторной

Ниже приведён краткий порядок выполнения работы:

1. Установить Docker, kubectl, k3d и Git.
